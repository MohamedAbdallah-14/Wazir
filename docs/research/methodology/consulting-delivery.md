# Consulting Delivery Methodologies — Deep Research

## Universal Pattern
Every serious firm has a structured pre-build phase. Names differ, function is identical: force clarity about problem and solution before committing engineering resources.

## ThoughtWorks
- **Discovery** → **Inception** (1-2 week workshop, all stakeholders)
- Lean Inception (Paulo Caroli): compressed to 5 days
- Never blend Discovery (find right problem) with Inception (plan right solution)
- Tech Radar as lightweight governance

## Pivotal Labs (Tanzu Labs)
- **Balanced Team**: PM + Designer + Engineer, each paired with client counterpart
- **Discovery & Framing**: D asks "what is the problem?", F asks "what subset should we build first?"
- "I do, we do, you do" skill transfer model
- 50+ named practices in open-source catalog

## IDEO / Double Diamond
- Diamond 1: Discover (diverge) → Define (converge) = find right problem
- Diamond 2: Develop (diverge) → Deliver (converge) = build right solution
- Key: diverge-converge TWICE. Most teams skip Diamond 1.

## Basecamp Shape Up
- **Appetite** (time budget, NOT estimate): "starts with a number, ends with a design"
- **Shaping**: rough (fat marker sketches), solved (main elements thought through), bounded
- **Pitch**: Problem + Appetite + Solution + Rabbit Holes + No-Gos
- **Betting Table** (not backlog): unbet pitches are discarded
- **Hill Charts**: uphill = resolving unknowns, downhill = execution
- No backlog. If not bet on, it dies.

## Spotify
- Squads (6-12) → Tribes (40-100) → Chapters (discipline) → Guilds (interest)
- **DIBB**: Data → Insights → Beliefs → Bets
- Autonomy vs alignment through cascading DIBB

## Amazon Working Backwards
- Write press release BEFORE building. If you can't write it, you don't understand it.
- PR-FAQ: Press Release (1 page) + External FAQs + Internal FAQs
- Silent reading (15-20 min) then discussion (40 min)
- "Truth-seeking, NOT selling"

## Stripe
- Writing culture as engineering infrastructure (CEO emails have footnotes)
- 20-page API design documents, strict review process
- Continuous planning, incremental delivery
- At scale, centralized API review became friction → pivot to education

## What Fails
- BDUF (Big Design Up Front): 200-page spec nobody reads
- NDUF (No Design Up Front): "we're agile, we don't plan"
- Sweet spot: EDUF (Enough Design Up Front)

## For AI Agents
- Appetite-based scoping is strongest fit (time box, not estimate)
- PR-FAQ's clarity-forcing function is critical for agents
- Pre-build phases remain fundamentally human (understanding users, strategic trade-offs)
- Build phase is where agents add most value
