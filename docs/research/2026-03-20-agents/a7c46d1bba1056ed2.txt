missing tests, architectural antipatterns\n\n**Weaknesses**:\n- Cannot see what it hasn't been told about (limited to loaded expertise)\n- Same model that may have been used for implementation (shared blind spots)\n- Pattern matching misses novel issues\n- Susceptible to the \"curse of expertise\" -- checks everything the library says but misses things the library doesn't cover\n\n**Effective dimensions**: Quality, Tests, Wiring, Drift (pattern-matchable against spec)\n\n#### Tier 2 (Codex / External) -- The Fresh-Eyes Outsider\n\n**What it checks**: What a smart engineer would notice reading the code cold, with no prior context about how it was built or what the expertise library says.\n\n**Strengths**:\n- Different model architecture (OpenAI vs Anthropic) -- genuinely different blind spots\n- No shared context with implementation -- cannot be \"anchored\" by the implementer's reasoning\n- Good at: logic errors, missing edge cases, integration issues, things that \"feel wrong\" but don't match a named antipattern\n- Reads the actual diff, not a summary -- catches things the internal reviewer might skim\n\n**Weaknesses**:\n- No expertise modules loaded (doesn't know your stack's specific pitfalls)\n- Slower, more expensive\n- May flag things the internal reviewer already checked and passed (noise)\n- Less consistent (different prompt, different model behavior)\n\n**Effective dimensions**: Correctness, Completeness, Intent Alignment, Scope Coverage (judgment-heavy, not pattern-matchable)\n\n#### The Critical Asymmetry\n\nTier 1 is a **known-unknowns detector** -- it checks for issues in a predefined catalogue. Tier 2 is an **unknown-unknowns detector** -- it reads with fresh eyes and catches things not in any catalogue. This is why Tier 1 runs first: clearing known issues means Tier 2 reviews clean code and can focus on novel concerns rather than wasting attention on things a linter could catch.\n\n---\n\n### 4. How Tier 1 Findings Feed Into Tier 2\n\n**Current mechanism**: Tier 1 findings are fixed before Tier 2 runs. Tier 2 never sees Tier 1's findings -- it sees the cleaned-up code. This is by design: Tier 2 reviews clean code, not a list of known issues.\n\n**What Tier 2 receives**:\n- The diff (committed or uncommitted changes)\n- Acceptance criteria from the spec\n- A dimension-specific prompt\n- No expertise modules, no Tier 1 findings, no implementation history\n\n**What Tier 2 does NOT receive**:\n- Tier 1's findings list\n- Expertise module content\n- Implementation context or reasoning\n\nThis is correct. Giving Tier 2 the Tier 1 findings would anchor it -- the fresh-eyes value comes precisely from not knowing what Tier 1 checked. This matches CodeRabbit's model (AI review is independent of human review) and Google's model (readability reviewer doesn't read the LGTM reviewer's comments before reviewing).\n\n---\n\n### 5. Conflict Resolution: When Tier 1 and Tier 2 Disagree\n\nThere are three disagreement scenarios:\n\n**Scenario A: Tier 1 passed, Tier 2 flags an issue.**\nThis is the normal case -- Tier 2's value is catching things Tier 1 missed. The finding is tagged `[Codex]` and treated as new. If blocking, the executor fixes and the fix goes through an internal regression check (Tier 1 re-runs to verify the fix didn't introduce new problems). This is already implemented in the \"Fix Cycle (Codex Findings)\" section of the reviewer skill.\n\n**Scenario B: Tier 1 flagged and fixed an issue, Tier 2 re-flags the same area.**\nThis means Tier 1's fix was insufficient or introduced a new problem. The finding is tagged `[Codex]`, and the fix cycle runs again. The `finding_hash` in `state.sqlite` distinguishes the original Tier 1 finding from the new Tier 2 finding (different description = different hash).\n\n**Scenario C: Tier 2 flags something the reviewer (the entity running both tiers) considers a false positive.**\nThe reviewer skill says: \"if you disagree with a Codex finding, note it with your rationale.\" This is the existing conflict resolution -- the internal reviewer (running the loop) is the arbiter. It can override a Codex finding by documenting rationale. This matches CodeRabbit's \"human wins, AI learns\" model. The finding is still persisted to `state.sqlite` with source `'codex'` -- if it recurs across multiple runs, the learning pipeline will surface it for user review.\n\n**Missing today**: There is no explicit protocol for the reverse -- Codex approving something that Tier 1 flagged. This doesn't arise in practice because Tier 1 findings must be fixed before Tier 2 runs. Tier 2 only reviews code that Tier 1 has already approved.\n\n---\n\n### 6. Security-Sensitive Review Escalation\n\nWhen `detectSecurityPatterns()` from `tooling/src/checks/security-sensitivity.js` triggers on the diff (matches patterns like `auth`, `token`, `sql`, `secret`, `jwt`, etc.), 6 additional security dimensions are added to the standard review:\n\n1. Injection (SQL, command, template, header)\n2. Authentication bypass\n3. Data exposure (PII, secrets, tokens in logs/responses)\n4. CSRF / SSRF\n5. XSS (stored, reflected, DOM)\n6. Secrets leakage (hardcoded keys, env vars in client code)\n\nThese are checked by both tiers -- Tier 1 with security expertise modules loaded (from `concerns.security-*` in composition-map), Tier 2 by Codex with security-focused prompting.\n\n---\n\n### 7. Summary: The Architecture as It Stands\n\n```\nCode Change\n  |\n  v\nTier 1: Internal Review (Sonnet, expertise-loaded)\n  - Loads expertise modules from composition-map.yaml\n  - Checks phase-specific dimensions (5 for task, 9 for final, etc.)\n  - Adds 6 security dims if security patterns detected\n  - Fix cycle: blocking findings -> executor fixes -> re-review\n  - Output: findings tagged [Internal]\n  |\n  | (only after Tier 1 is clean -- no blocking findings)\n  v\nTier 2: External Review (Codex, fresh eyes)\n  - No expertise modules, no Tier 1 context\n  - Reviews clean code against acceptance criteria\n  - Output: findings tagged [Codex]\n  - Conflict: reviewer arbitrates, documents rationale\n  |\n  v\nMerged Findings\n  - All findings persisted to state.sqlite with source attribution\n  - finding_hash enables cross-run recurrence detection\n  - Recurring findings -> learning candidates -> user-approved learnings\n  - Learnings injected into future executor context (prevention > detection)\n```\n\n**Model selection rationale**: Sonnet for Tier 1 because it's pattern matching against loaded expertise (comprehension task).
