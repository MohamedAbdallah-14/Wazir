This is defense in depth.\n\n---\n\n## Synthesis: Universal Patterns Across All Three\n\n| Pattern | CrewAI | LangGraph | GitHub Actions |\n|---------|--------|-----------|----------------|\n| **Who holds the loop** | Framework (Python for-loop) | Framework (checkpointer + graph runtime) | Platform (branch protection engine) |\n| **Agent can skip?** | No -- never sees guardrail code | No -- execution literally paused | No -- merge API returns 403 |\n| **Feedback mechanism** | String feedback appended to agent context | Resume value becomes interrupt return | Check status red/green in PR UI |\n| **Retry limit** | `guardrail_max_retries` (default 3) | Unlimited (while loop + interrupt) | Unlimited (push new commits) |\n| **State persistence** | In-memory task context | Checkpointer (SQLite, memory, etc.) | GitHub's check run database |\n| **Escape hatch** | Max retries exhausted = last output used | Human provides resume value | Admin bypass on branch protection |\n\n---\n\n## Wazir Phase Guardrail Specifications\n\nBased on the existing Wazir architecture (manifest phases: `init` -> `clarifier` -> `executor` -> `final_review`; workflows within each phase; existing guards in `tooling/src/guards/`), here are the guardrail function specs per workflow boundary.\n\n### Design Principles (derived from research)\n\n1. **Framework holds the loop.** The `/wazir` controller skill calls each subagent and validates output before advancing.
