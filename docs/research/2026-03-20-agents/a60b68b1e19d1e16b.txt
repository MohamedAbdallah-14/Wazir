Depth-Scaled Frequency Rules\n\n| Signal Type | Quick | Standard | Deep |\n|---|---|---|---|\n| Phase map (Pattern A) | Phase transitions only | Workflow transitions | Workflow transitions + sub-step |\n| Finding updates (Pattern B) | Key decisions only | Every decision | Every decision + alternatives |\n| Artifact previews (Pattern C) | None | 1 per major artifact | 2 per major artifact |\n| Time estimates (Pattern D) | Phase-level only | Phase + recalibration | Phase + per-workflow + recalibration |\n| Heartbeat (Pattern E) | Every 3 min | Every 2 min | Every 90 sec |\n| Completion proof | Compact (3 lines) | Standard (as shown above) | Extended (include counterfactuals) |\n\n---\n\n## 6. Key Design Principles Extracted\n\n1. **Separate heartbeat from insight.** Heartbeats prove liveness. Findings prove value. Never conflate them. (Devin thread model)\n\n2. **Progressive disclosure, not progressive overload.** Show the summary in the conversation; put the detail in the artifact files. (GitHub step summary vs. logs)\n\n3. **Spotlight the active step.** The user should never have to search for what is happening now. Auto-focus on the current workflow. (Buildkite Follow mode)\n\n4. **Completion proofs need counterfactuals.** \"Phase complete\" is not proof of value. \"Without this phase, X would have gone wrong\" is. (Devin session insights -- \"misleading knowledge\" tracking)\n\n5. **Recalibrate, do not hide delays.** When a time estimate drifts, say so immediately with a reason. Silence during overrun destroys trust faster than an honest \"this is taking longer because...\" (GitLab ETA research)\n\n6. **The final artifact is the proof.** PRs, test results, committed code. Not summaries of what the agent claims it did.
