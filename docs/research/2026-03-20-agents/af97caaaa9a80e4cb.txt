There is no runtime validation that a key was populated -- if Node B accesses `state[\"research_brief\"]` and Node A didn't populate it, you get a `KeyError` at runtime. The enforcement is structural (graph edges guarantee execution order) not schematic.\n\n### 3. GitHub Actions `needs.<job>.outputs`\n\n**How it works:** Three-layer mechanism:\n1. **Step sets output:** `echo \"key=value\" >> $GITHUB_OUTPUT`\n2. **Job declares output:** `jobs.<id>.outputs.<name>: ${{ steps.<step_id>.outputs.<key> }}`\n3. **Downstream job reads:** `needs: [build]` + `${{ needs.build.outputs.<name> }}`\n\n**Dependency enforcement:** The `needs:` keyword creates a hard DAG edge. The downstream job will not start until all `needs` jobs succeed. If a needed job fails or is skipped, the dependent job is skipped (unless overridden with `if: always()`).\n\n**What's NOT enforced:** There is no schema validation on outputs. Outputs are always strings. If a job declares an output but the step never sets it, the value is empty string -- no error. The downstream job gets `\"\"` and must handle it. Size limit: 1MB per output, 50MB total per workflow. For larger data, you use artifacts (upload/download pattern).\n\n**Key insight:** The enforcement is *execution ordering* (DAG edges via `needs`), not *content validation*. The outputs are untyped strings. You can add `if: needs.build.outputs.result != ''` as a manual guard, but the framework doesn't require it.\n\n### 4. Temporal Activity Results\n\n**How it works:** Activities are regular functions with typed signatures. The workflow orchestrator calls them sequentially with `await`:\n\n```python\nresult1 = await workflow.execute_activity(\n    research_activity,          # typed: (ResearchInput) -> ResearchBrief\n    ResearchInput(topic=\"...\"),\n    start_to_close_timeout=timedelta(minutes=5),\n)\n\nresult2 = await workflow.execute_activity(\n    spec_activity,              # typed: (ResearchBrief) -> SpecArtifact\n    result1,                    # compiler enforces type match\n    start_to_close_timeout=timedelta(minutes=10),\n)\n```\n\n**Enforcement model:** This is *function-parameter-level* enforcement. The Python/Go/Java type system enforces that `result1` (type `ResearchBrief`) is a valid argument to `spec_activity` (which expects `ResearchBrief`). Temporal recommends single Pydantic model or dataclass parameters for both inputs and outputs, with the Pydantic data converter handling serialization/deserialization.\n\n**Key insight:** This is the strongest enforcement model. The dependency is literally a function parameter -- it can't compile/run if the types don't match. The artifact is not a file on disk; it's a typed value in the workflow's execution history, durably stored and replayable.\n\n---\n\n### Comparison Matrix\n\n| Framework | Dependency Declaration | Type Enforcement | Validation Point | Failure Mode |\n|---|---|---|---|---|\n| **CrewAI** | `context=[task_ref]` | Pydantic model on `output_pydantic` | On write (output validation) | Retry or exception |\n| **LangGraph** | Graph edges + shared typed state | TypedDict/Pydantic annotations | On read (KeyError if missing) | Runtime crash |\n| **GitHub Actions** | `needs: [job_id]` | None (strings only) | Never (empty string is valid) | Silent empty value |\n| **Temporal** | Function parameter passing | Language type system + Pydantic converter | On call (type mismatch) | Compile/runtime type error |\n\n---\n\n### Synthesis for Wazir: The Design\n\n**Current state:** Wazir has JSON schemas per artifact, workflow markdown files listing inputs/outputs in prose, and `phase_prerequisites` in the manifest that list file paths. Subagents: The Handoff Protocol\n\nThe recommended pattern combines all four frameworks' strengths:\n\n**1. Write-time validation (from CrewAI):**\nWhen a phase completes, it writes its artifact to `~/.wazir/projects/<slug>/runs/<run-id>/artifacts/<phase>.json`.
