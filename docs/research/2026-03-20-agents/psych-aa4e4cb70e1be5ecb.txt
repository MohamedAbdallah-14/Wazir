The system decides — the user does NOT pick.\n   257→\n   258→| Class | Workflows | Rules |\n   259→|-------|-----------|-------|\n   260→| **Core** (always run) | `clarify`, `execute`, `verify`, `review` | Never skipped |\n   261→| **Adaptive** (run when evidence says so) | `discover`, `design`, `author`, `specify` | Skipped for bugfix/docs/spike at quick depth |\n   262→| **Scale** (intensity varies) | `spec-challenge`, `plan-review`, `design-review` | Loop cap controls iteration depth |\n   263→| **Post-run** (always run) | `learn`, `prepare_next` | Part of Final Review phase |\n   264→\n   265→Log skip decisions with reasons in `workflow_policy`.\n   266→\n   267→### Confidence Gate\n   268→\n   269→After building run config:\n   270→\n   271→- **High confidence** — one-line summary and proceed:\n   272→  > **Running: standard depth, feature, sequential. Proceeding...**\n   273→\n   274→- **Low confidence** — show plan and ask:\n   275→\n   276→  Ask the user via AskUserQuestion:\n   277→  - **Question:** \"Does this run configuration look right?\"\n   278→  - **Options:**\n   279→    1. \"Yes, proceed\" *(Recommended)*\n   280→    2. \"No, let me adjust\"\n   281→\n   282→  Wait for the user's selection before continuing.\n   283→\n   284→```bash\n   285→wazir capture event --run <run-id> --event phase_exit --phase init --status completed\n   286→```\n   287→\n   288→Run the phase report and display it to the user:\n   289→```bash\n   290→wazir report phase --run <run-id> --phase init\n   291→```\n   292→\n   293→Output the report content to the user in the conversation.\n   294→\n   295→---\n   296→\n   297→# Interaction Modes\n   298→\n   299→The `interaction_mode` field in run-config controls how the pipeline interacts with the user:\n   300→\n   301→| Mode | Inline modifier | Behavior | Best for |\n   302→|------|----------------|----------|----------|\n   303→| **`guided`** | (default) | Pipeline runs, pauses at phase checkpoints for user approval. Current default behavior. | Most work |\n   304→| **`auto`** | `/wazir auto ...` | No human checkpoints. Codex reviews all. Gating agent decides continue/loop_back/escalate. Stops ONLY on escalate. | Overnight, clear spec, well-understood domain |\n   305→| **`interactive`** | `/wazir interactive ...` | More questions, more discussion, co-designs with user. Researcher presents options. Executor checks approach before coding. | Ambiguous requirements, new domain, learning |\n   306→\n   307→## `auto` mode constraints\n   308→\n   309→- **Codex REQUIRED** — refuse to start auto mode if `multi_tool.codex` is not configured in `.wazir/state/config.json`. Error: \"Auto mode requires an external reviewer (Codex). Configure it first or use guided mode.\"\n   310→- **On escalate:** STOP immediately, write the escalation reason to `.wazir/runs/<id>/escalations/`, and wait for user input\n   311→- **Wall-clock limit:** default 4 hours. If exceeded, stop with escalation.\n   312→- **Never auto-commits to main** — always work on feature branch\n   313→- All checkpoints (AskUserQuestion) are skipped — gating agent evaluates phase reports and decides\n   314→\n   315→## `guided` mode (default)\n   316→\n   317→Current behavior — no changes needed. Checkpoints at phase boundaries, user approves before advancing.\n   318→\n   319→## `interactive` mode\n   320→\n   321→- **Clarifier:** asks more detailed questions, presents research findings with options: \"I found 3 approaches — which interests you?\"\n   322→- **Executor:** checks approach before coding: \"I'm about to implement auth with Supabase — sound right?\"\n   323→- **Reviewer:** discusses findings with user, not just presents verdict: \"I found a potential auth bypass — here's why I think it's high severity, do you agree?\"\n   324→- Slower but highest quality for complex/ambiguous work\n   325→\n   326→## Mode checking in phase skills\n   327→\n   328→All phase skills check `interaction_mode` from run-config at every checkpoint:\n   329→\n   330→```\n   331→# Read from run-config\n   332→interaction_mode = run_config.interaction_mode ?? 'guided'\n   333→\n   334→# At each checkpoint:\n   335→if interaction_mode == 'auto':\n   336→    # Skip checkpoint, let gating agent decide\n   337→elif interaction_mode == 'interactive':\n   338→    # More detailed question, present options, discuss\n   339→else:\n   340→    # guided — standard checkpoint with AskUserQuestion\n   341→```\n   342→\n   343→---\n   344→\n   345→# Two-Level Phase Model\n   346→\n   347→The pipeline has 4 top-level **phases**, each containing multiple **workflows** with review loops:\n   348→\n   349→```\n   350→Phase 1: Init\n   351→  └── (inline — controller handles directly)\n   352→\n   353→Phase 2: Clarifier        → dispatched as SUBAGENT\n   354→  ├── discover (research) ← research-review loop\n   355→  ├── clarify ← clarification-review loop\n   356→  ├── specify ← spec-challenge loop\n   357→  ├── author (adaptive) ← approval gate\n   358→  ├── design ← design-review loop\n   359→  └── plan ← plan-review loop\n   360→\n   361→Phase 3: Executor          → dispatched as SUBAGENT\n   362→  ├── execute (per-task) ← task-review loop per task\n   363→  └── verify\n   364→\n   365→Phase 4: Final Review      → dispatched as SUBAGENT\n   366→  ├── review (final) ← scored review\n   367→  ├── learn\n   368→  └── prepare_next\n   369→```\n   370→\n   371→**Event capture uses both levels.** When emitting phase events, include `--parent-phase`:\n   372→```bash\n   373→wazir capture event --run <id> --event phase_enter --phase discover --parent-phase clarifier --status in_progress\n   374→```\n   375→\n   376→**Progress markers between workflows:** After each workflow completes, output:\n   377→> Phase 2: Clarifier > Workflow: specify (3 of 6 workflows complete)\n   378→\n   379→**`wazir status` shows both levels:** \"Phase 2: Clarifier > Workflow: specify\"\n   380→\n   381→---\n   382→\n   383→# Subagent Controller Architecture\n   384→\n   385→**This is the core enforcement mechanism.** The controller (this skill, wz:wazir) dispatches ONE fresh Agent per phase. Each subagent gets a clean 200K context with only its skill instructions and artifact paths — never the full pipeline context.\n   386→\n   387→## Why Subagents\n   388→\n   389→A single-context pipeline allows the agent to rationalize skipping phases (\"the input is clear enough\"). Subagent isolation prevents this:\n   390→- Each subagent ONLY sees its own phase instructions\n   391→- No subagent can see or skip another phase\n   392→- The controller validates artifacts BETWEEN phases\n   393→- Hooks provide a second enforcement layer independent of prompt compliance\n   394→\n   395→## Controller Loop\n   396→\n   397→```\n   398→initialize pipeline-state.json via createPipelineState(runId, stateRoot)\n   399→transitionPhase(stateRoot, 'clarify')\n   400→\n   401→for each phase in [clarify, execute, review]:\n   402→  1. Update pipeline-state.json: current_phase = phase\n   403→  2. Run pre-phase guardrail (validate previous phase artifacts)\n   404→  3. Build subagent prompt (see Subagent Prompt Template below)\n   405→  4. Dispatch: Agent(prompt=..., description=\"wazir: <phase>\", mode=\"bypassPermissions\")\n   406→  5. On completion: validate output artifacts via runGuardrail(phase, state, runDir)\n   407→  6. If guardrail passes:\n   408→     a. completePhase(stateRoot, phase, artifacts)\n   409→     b. Continue to next phase\n   410→  7. If guardrail fails: execute Retry Ladder\n   411→  8. Capture events:\n   412→     wazir capture event --run <id> --event phase_exit --phase <phase> --status completed\n   413→\n   414→transitionPhase(stateRoot, 'complete')\n   415→```\n   416→\n   417→**CRITICAL: No phase runs inline in the controller.** The controller ONLY:\n   418→- Manages state transitions\n   419→- Dispatches subagents\n   420→- Validates guardrails\n   421→- Handles retry/escalation\n   422→- Presents results to the user\n   423→\n   424→## Subagent Prompt Template\n   425→\n   426→Each subagent receives this prompt structure:\n   427→\n   428→```\n   429→You are running the {PHASE} phase of the Wazir pipeline.\n   430→\n   431→Run ID: {run_id}\n   432→Run directory: {run_dir}\n   433→State root: {state_root}\n   434→Depth: {depth}\n   435→Interaction mode: {interaction_mode}\n   436→\n   437→## Your Instructions\n   438→{Read and paste the full content of skills/{phase_skill}/SKILL.md here}\n   439→\n   440→## Input Artifacts (read from disk)\n   441→{List of file paths the subagent should read as input}\n   442→\n   443→## Output Artifacts (write to disk)\n   444→{List of file paths the subagent must produce}\n   445→\n   446→## Rules\n   447→- Read your input artifacts from the paths above\n   448→- Write your output artifacts to the paths above\n   449→- Do NOT skip any step in your instructions\n   450→- Use wazir index for codebase exploration\n   451→- Use context-mode for large command outputs\n   452→- When done, state which artifacts you produced\n   453→```\n   454→\n   455→The controller reads the phase skill from disk and includes it in the prompt. This ensures each subagent has the latest skill version.\n   456→\n   457→## Subagent Dispatch Rules\n   458→\n   459→1. **No nesting** — all subagents dispatched at depth=1 from the controller\n   460→2. **No context sharing** — subagents communicate only via artifacts on disk\n   461→3. **No pipeline state awareness** — subagents don't read pipeline-state.json\n   462→4. **Controller reads skills** — Read `skills/{name}/SKILL.md` before dispatch, paste into prompt\n   463→5. **Verify phase handled by executor** — the executor subagent handles both execute + verify workflows\n   464→\n   465→## Retry Ladder\n   466→\n   467→If a guardrail fails after a phase subagent completes:\n   468→\n   469→```\n   470→retry_count = 0\n   471→while guardrail fails:\n   472→  retry_count++\n   473→  if retry_count <= 2:\n   474→    # Re-dispatch same phase with failure feedback\n   475→    prompt += \"\\n\\nPREVIOUS ATTEMPT FAILED GUARDRAIL:\\n{guardrail.reason}\\nMissing: {guardrail.missing}\\nFix these issues.\"\n   476→    Dispatch Agent again\n   477→  elif retry_count == 3:\n   478→    # Escalate model (use Opus if not already)\n   479→    prompt += \"\\n\\nESCALATED: Previous attempts failed. Produce ALL required artifacts.\"\n   480→    Dispatch Agent with model=\"opus\"\n   481→  else:\n   482→    # Escalate to human\n   483→    Ask user: \"Phase {phase} failed guardrail after {retry_count} attempts: {reason}\"\n   484→    Options: 1. Retry manually  2. Skip phase  3. Abort run\n   485→    break\n   486→```\n   487→\n   488→## Pipeline State Management\n   489→\n   490→The controller manages `pipeline-state.json` at `$STATE_ROOT/pipeline-state.json`:\n   491→\n   492→```javascript\n   493→// Before first phase\n   494→createPipelineState(runId, stateRoot)\n   495→transitionPhase(stateRoot, 'clarify')\n   496→\n   497→// Between phases\n   498→transitionPhase(stateRoot, 'execute')\n   499→\n   500→// After each phase\n   501→completePhase(stateRoot, phase, { artifactName: { path: '...' } })\n   502→\n   503→// When done\n   504→transitionPhase(stateRoot, 'complete')\n   505→```\n   506→\n   507→The Stop hook reads this file to block premature completion.\n   508→The PreToolUse hook reads this file to enforce phase-specific tool restrictions.\n   509→\n   510→---\n   511→\n   512→# Phase 2: Clarifier (Subagent)\n   513→\n   514→**Before dispatching, output to the user:**\n   515→\n   516→> **Clarifier Phase** — Dispatching clarifier subagent to research your codebase, clarify requirements, harden the spec, brainstorm designs, and produce an execution plan.\n   517→>\n   518→> **Why this matters:** Without this, I'd guess your tech stack, misunderstand constraints, miss edge cases in the spec, and build the wrong architecture.
