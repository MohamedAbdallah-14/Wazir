No role reviews its own output.\n    10→\n    11→```\n    12→Producer emits artifact\n    13→  -> Reviewer runs review loop (N passes, Codex if available)\n    14→  -> Findings returned to producer\n    15→  -> Producer fixes and resubmits\n    16→  -> Loop until all passes exhausted or cap reached\n    17→  -> Escalate to user if cap exceeded\n    18→```\n    19→\n    20→When Codex is available, the reviewer role delegates to `codex review` as a secondary input while maintaining its own independent primary verdict.\n    21→\n    22→---\n    23→\n    24→## Per-Task Review vs Final Review\n    25→\n    26→These are two structurally different constructs:\n    27→\n    28→| | Per-Task Review | Final Review |\n    29→|---|---|---|\n    30→| **When** | During execution, after each task | After all execution + verification complete |\n    31→| **Dimensions** | 5 task-execution dims (correctness, tests, wiring, drift, quality) | 9 scored dims (correctness, completeness, wiring, verification, drift, quality, documentation, intent alignment, scope coverage) |\n    32→| **Scope** | Single task's uncommitted changes | Entire implementation vs spec/plan |\n    33→| **Output** | Pass/fix loop, no score | Scored verdict (0-90), PASS/FAIL |\n    34→| **Workflow** | Inline in execution flow | `workflows/review.md` |\n    35→| **Skill** | `wz:reviewer` in `task-review` mode | `wz:reviewer` in `final` mode |\n    36→| **Log filename** | `<phase>-task-<NNN>-review-pass-<N>.md` | `final-review.md` |\n    37→\n    38→---\n    39→\n    40→## Standalone Mode\n    41→\n    42→When no `.wazir/runs/latest/` directory exists (standalone skill invocation outside a pipeline run):\n    43→\n    44→1. **Review loops still run** -- the review logic is embedded in the skill, not dependent on run state.\n    45→2. **Artifact location** -- artifacts live in `docs/plans/`. This is the canonical standalone artifact path.\n    46→3. **Review log location** -- review logs go alongside the artifact: `docs/plans/YYYY-MM-DD-<topic>-review-pass-<N>.md`. No temp dir.\n    47→4. **Loop cap is SKIPPED entirely** -- no `wazir capture loop-check` call. The loop runs for exactly `pass_counts[depth]` passes (3/5/7) and stops. No cap guard, no fallback constant.\n    48→5. **`wazir capture loop-check`** -- not invoked in standalone mode. The standalone detection happens before the cap guard call.\n    49→\n    50→Detection logic:\n    51→\n    52→```\n    53→if .wazir/runs/latest/ exists:\n    54→  run_mode = \"pipeline\"\n    55→  log_dir = .wazir/runs/latest/reviews/\n    56→  cap_guard = wazir capture loop-check (full guard)\n    57→else:\n    58→  run_mode = \"standalone\"\n    59→  artifact_dir = docs/plans/\n    60→  log_dir = docs/plans/  (alongside artifact)\n    61→  cap_guard = none (depth pass count is the only limit)\n    62→```\n    63→\n    64→---\n    65→\n    66→## Review Loop Pseudocode\n    67→\n    68→```\n    69→review_loop(artifact_path, phase, dimensions[], depth, config, options={}):\n    70→\n    71→  # options.mode      -- explicit review mode (required)\n    72→  # options.task_id   -- task identifier for task-scoped reviews (optional)\n    73→\n    74→  # Standalone detection\n    75→  run_mode = detect_run_mode()  # \"pipeline\" or \"standalone\"\n    76→\n    77→  # Fixed pass counts -- no extension\n    78→  pass_counts = { quick: 3, standard: 5, deep: 7 }\n    79→  total_passes = pass_counts[depth]\n    80→\n    81→  # Depth-aware dimension subsets (coverage contract)\n    82→  depth_dimensions = {\n    83→    quick:    dimensions[0:3],     # first 3 dimensions only\n    84→    standard: dimensions[0:5],     # first 5\n    85→    deep:     dimensions,          # all available\n    86→  }\n    87→  active_dims = depth_dimensions[depth]\n    88→\n    89→  codex_available = check_codex()  # which codex && codex --version\n    90→\n    91→  for pass_number in 0..total_passes-1:\n    92→\n    93→    # --- Cap guard check (pipeline mode only, before each pass) ---\n    94→    if run_mode == \"pipeline\":\n    95→      loop_check_args = \"--run <run-id> --phase <phase> --loop-count <pass_number+1>\"\n    96→      if options.task_id:\n    97→        loop_check_args += \" --task-id <task_id>\"\n    98→      wazir capture loop-check $loop_check_args\n    99→      # loop-check wraps: event capture + evaluateLoopCapGuard\n   100→      # If loop_cap_guard fires (exit 43), stop immediately:\n   101→      if last_exit_code == 43:\n   102→        log(\"Loop cap reached for phase: <phase>. Escalating to user.\")\n   103→        escalate_to_user(evidence_gathered_so_far)\n   104→        return { pass_count: pass_number, escalated: true }\n   105→    # Standalone mode: no cap guard. Loop runs for total_passes and stops.\n   106→\n   107→    dimension = active_dims[pass_number % len(active_dims)]\n   108→\n   109→    # --- Primary review (reviewer role, not producer) ---\n   110→    # Mode is always explicit -- passed by caller via options.mode\n   111→    findings = self_review(artifact_path, focus=dimension, mode=options.mode)\n   112→\n   113→    # --- Secondary review (Codex, if available) ---\n   114→    if codex_available:\n   115→      codex_exit_code, codex_output = run_codex_review(artifact_path, dimension)\n   116→      if codex_exit_code != 0:\n   117→        # Codex failed -- log error, fall back to self-review for this pass\n   118→        log_error(\"Codex exited \" + codex_exit_code + \": \" + codex_output.stderr)\n   119→        mark_pass_codex_unavailable(pass_number)\n   120→        # Do NOT treat Codex failure as clean. Self-review findings stand alone.\n   121→      else:\n   122→        codex_findings = parse(codex_output.stdout)\n   123→        merge(findings, codex_findings, preserve_attribution=true)\n   124→\n   125→    # --- Log the review pass ---\n   126→    if run_mode == \"pipeline\":\n   127→      if options.task_id:\n   128→        log_path = .wazir/runs/latest/reviews/<phase>-task-<task_id>-review-pass-<N>.md\n   129→      else:\n   130→        log_path = .wazir/runs/latest/reviews/<phase>-review-pass-<N>.md\n   131→      log(pass_number+1, dimension, findings) -> log_path\n   132→    else:\n   133→      log_path = docs/plans/YYYY-MM-DD-<topic>-review-pass-<N>.md\n   134→      log(pass_number+1, dimension, findings) -> log_path\n   135→\n   136→    if findings.has_issues:\n   137→      # --- Fix and re-submit (MANDATORY) ---\n   138→      # The producer MUST fix findings and the reviewer MUST re-review.\n   139→      # \"Fix and continue without re-review\" is EXPLICITLY PROHIBITED.\n   140→      producer_fix(artifact_path, findings)\n   141→      # Continue to next pass -- the fix will be re-reviewed\n   142→\n   143→  # --- Post-loop: escalation if issues remain ---\n   144→  if remaining.has_issues:\n   145→    # Cap reached with unresolved findings. Present to user:\n   146→    # 1. Approve with known issues (Recommended if non-blocking)\n   147→    # 2. Fix manually and re-run\n   148→    # 3. Abort\n   149→    escalate_to_user(remaining, options=[\n   150→      \"approve-with-issues\",\n   151→      \"fix-manually-and-rerun\",\n   152→      \"abort\"\n   153→    ])\n   154→    # User decides. If approved, log \"user-approved-with-issues\" in final pass file.\n   155→\n   156→  return { pass_count: total_passes, issues_found, issues_fixed, remaining, attributions }\n   157→```\n   158→\n   159→Key properties of this pseudocode:\n   160→\n   161→1. **Fixed pass counts** -- Quick is exactly 3, standard exactly 5, deep exactly 7. No `max_passes = min_passes + 3`. No clean-streak early-exit. No extension.\n   162→2. **Task-scoped log filenames** -- `<phase>-task-<NNN>-review-pass-<N>.md` for per-task reviews, preventing log clobbering in parallel mode.\n   163→3. **Task-scoped loop cap keys** -- `--task-id` flag on `loop-check` so each task gets its own counter in `phase_loop_counts`.\n   164→4. **Explicit review mode** -- `options.mode` is always passed by the caller. No auto-detection.\n   165→5. **Codex error handling** -- non-zero exit is logged, pass marked `codex-unavailable`, self-review findings used alone. Never treated as clean.\n   166→6. **Standalone mode** -- uses `docs/plans/` for artifacts and logs. No temp dir. No cap guard at all.\n   167→\n   168→---\n   169→\n   170→## Codex Error Handling Contract\n   171→\n   172→```\n   173→run_codex_review(artifact_path, dimension):\n   174→  CODEX_MODEL = read_config('.wazir/state/config.json', '.multi_tool.codex.model') or \"gpt-5.4\"\n   175→\n   176→  if is_code_artifact:\n   177→    cmd = codex review -c model=\"$CODEX_MODEL\" --uncommitted --title \"...\" \"Review for [dimension]...\"\n   178→    # or: codex review -c model=\"$CODEX_MODEL\" --base <sha> for committed changes\n   179→  else:\n   180→    cmd = cat <artifact_path> | codex exec -c model=\"$CODEX_MODEL\" \"Review this [type] for [dimension]...\"\n   181→\n   182→  result = execute(cmd, timeout=120s, capture_stderr=true)\n   183→\n   184→  if result.exit_code != 0:\n   185→    return (result.exit_code, { stderr: result.stderr, stdout: \"\" })\n   186→    # Caller handles: log error, mark codex-unavailable, use self-review only\n   187→\n   188→  return (0, { stdout: result.stdout, stderr: result.stderr })\n   189→```\n   190→\n   191→Rules:\n   192→\n   193→- If Codex exits non-zero, log the full stderr.\n   194→- Mark the pass as `codex-unavailable` in the review log metadata.\n   195→- Fall back to self-review for that pass only. Do not skip the pass.\n   196→- Do not retry Codex on the same pass. If Codex fails on pass 2, pass 3 still tries Codex (transient failures recover).\n   197→- Never treat a Codex failure as a clean review pass.\n   198→\n   199→---\n   200→\n   201→## Codex Availability Probe\n   202→\n   203→Before any Codex call, verify availability once at loop start:\n   204→\n   205→```bash\n   206→which codex >/dev/null 2>&1 && codex --version >/dev/null 2>&1\n   207→```\n   208→\n   209→If the probe fails, set `codex_available = false` for the entire loop. Fall back to self-review only. Never error out.\n   210→\n   211→Per-invocation failures (Codex available but a single call fails) are handled separately by the error contract above.\n   212→\n   213→---\n   214→\n   215→## Codex Artifact-Scoped Review\n   216→\n   217→Never use `codex review` for non-code artifacts (specs, plans, designs). Instead, pipe the artifact content via stdin:\n   218→\n   219→```bash\n   220→CODEX_MODEL=$(jq -r '.multi_tool.codex.model // empty' .wazir/state/config.json 2>/dev/null)\n   221→CODEX_MODEL=${CODEX_MODEL:-gpt-5.4}\n   222→cat .wazir/runs/latest/clarified/spec-hardened.md | \\\n   223→  codex exec -c model=\"$CODEX_MODEL\" \"Review this specification for: [dimension]. Be specific, cite sections. Say CLEAN if no issues.\" \\\n   224→  2>&1 | tee .wazir/runs/latest/reviews/spec-challenge-review-pass-N.md\n   225→```\n   226→\n   227→For code artifacts, use `codex review -c model=\"$CODEX_MODEL\" --uncommitted` (or `--base` for committed changes). See the next section for details.\n   228→\n   229→---\n   230→\n   231→## Code Review Scoping\n   232→\n   233→**Rule: review BEFORE commit.**\n   234→\n   235→For each task during execution:\n   236→\n   237→1. Implement the task (changes are uncommitted).\n   238→2. Review the uncommitted changes using the **5 task-execution dimensions** (NOT the 9 final-review dimensions):\n   239→   ```bash\n   240→   CODEX_MODEL=$(jq -r '.multi_tool.codex.model // empty' .wazir/state/config.json 2>/dev/null)\n   241→CODEX_MODEL=${CODEX_MODEL:-gpt-5.4}\n   242→   codex review -c model=\"$CODEX_MODEL\" --uncommitted --title \"Task NNN: <summary>\" \\\n   243→     \"Review against acceptance criteria: <criteria>\" \\\n   244→     2>&1 | tee .wazir/runs/latest/reviews/execute-task-NNN-review-pass-N.md\n   245→   ```\n   246→3. Fix any findings (still uncommitted).\n   247→4. Re-review until all passes exhausted or cap reached.\n   248→5. **Only after review passes:** commit with conventional commit format.\n   249→\n   250→**If changes are already committed** (e.g., subagent workflow where the implementer subagent commits before review):\n   251→\n   252→```bash\n   253→# Capture the SHA before the task starts\n   254→PRE_TASK_SHA=$(git rev-parse HEAD)\n   255→\n   256→# ... subagent implements and commits ...\n   257→\n   258→# Review the committed changes against the pre-task baseline\n   259→CODEX_MODEL=$(jq -r '.multi_tool.codex.model // empty' .wazir/state/config.json 2>/dev/null)\n   260→CODEX_MODEL=${CODEX_MODEL:-gpt-5.4}\n   261→codex review -c model=\"$CODEX_MODEL\" --base $PRE_TASK_SHA --title \"Task NNN: <summary>\" \\\n   262→  \"Review against acceptance criteria: <criteria>\" \\\n   263→  2>&1 | tee .wazir/runs/latest/reviews/execute-task-NNN-review-pass-N.md\n   264→```\n   265→\n   266→---\n   267→\n   268→## Dimension Sets\n   269→\n   270→### Research Dimensions (5)\n   271→\n   272→1. **Coverage** -- all briefing topics researched\n   273→2. **Source quality** -- authoritative, current sources\n   274→3. **Relevance** -- research answers the actual questions\n   275→4. **Gaps** -- missing info that blocks later phases\n   276→5. **Contradictions** -- conflicting sources identified\n   277→\n   278→### Spec/Clarification Dimensions (5)\n   279→\n   280→1. **Completeness** -- all requirements covered\n   281→2. **Testability** -- each criterion verifiable\n   282→3. **Ambiguity** -- no dual-interpretation statements\n   283→4. **Assumptions** -- hidden assumptions explicit\n   284→5. **Scope creep** -- nothing beyond briefing\n   285→\n   286→### Design-Review Dimensions (5)\n   287→\n   288→Matches canonical `workflows/design-review.md`:\n   289→\n   290→1. **Spec coverage** -- does the design address every acceptance criterion with a visual component?\n   291→2. **Design-spec consistency** -- does the design introduce anything not in the spec? (scope creep check)\n   292→3. **Accessibility** -- color contrast ratios (WCAG 2.1 AA), focus states, touch target sizes (44x44px minimum)\n   293→4. **Visual consistency** -- design tokens form a coherent system, dark/light mode alignment\n   294→5. **Exported-code fidelity** -- do exported scaffolds match the designs? Mismatches are failures here, not implementation concerns.\n   295→\n   296→### Plan Dimensions (8)\n   297→\n   298→1. **Completeness** -- all design decisions mapped to tasks\n   299→2. **Ordering** -- dependencies correct, parallelizable identified\n   300→3. **Atomicity** -- each task fits one session\n   301→4. **Testability** -- concrete verification per task\n   302→5. **Edge cases** -- error paths covered\n   303→6. **Security** -- auth, injection, data exposure\n   304→7. **Integration** -- tasks connect end-to-end\n   305→8. **Input Coverage** -- every distinct item in the original input maps to at least one task. If `tasks < input items`, HIGH finding listing missing items\n   306→\n   307→### Task Execution Dimensions (5)\n   308→\n   309→Used for per-task review during execution:\n   310→\n   311→1. **Correctness** -- code matches spec\n   312→2. **Tests** -- real tests, not mocked/faked\n   313→3. **Wiring** -- all paths connected\n   314→4. **Drift** -- matches task spec\n   315→5. **Quality** -- naming, error handling\n   316→\n   317→### Final Review Dimensions (9)\n   318→\n   319→Used for `workflows/review.md` scored gate:\n   320→\n   321→1. **Correctness** -- does the code do what the spec says?\n   322→2. **Completeness** -- are all acceptance criteria met?\n   323→3. **Wiring** -- are all paths connected end-to-end?\n   324→4. **Verification** -- is there evidence (tests, type checks) for each claim?\n   325→5. **Drift** -- does the implementation match the approved plan?\n   326→6. **Quality** -- code style, naming, error handling, security\n   327→7. **Documentation** -- changelog entries, commit messages, comments\n   328→8. **Intent Alignment** -- does the implementation match what the user originally asked for, including mid-run corrections from `user-input-log.ndjson`? Compare against the RAW INPUT, not the spec. Score 0 if implementation contradicts a user correction.\n   329→9. **Scope Coverage** -- count distinct items in the original input vs items implemented. Flag missing items as HIGH. Score = (implemented / input items) * 10.\n   330→\n   331→The final review dimensions are the existing 9 from `skills/reviewer/SKILL.md`. `workflows/review.md` is not modified by this pattern.\n   332→\n   333→---\n   334→\n   335→## Per-Depth Coverage Contract\n   336→\n   337→| Depth | Research | Spec | Design-Review | Plan | Task Execution | Final Review |\n   338→|-------|----------|------|---------------|------|----------------|--------------|\n   339→| Quick | dims 1-3, 3 passes | dims 1-3, 3 passes | dims 1-3, 3 passes | dims 1-3, 3 passes | dims 1-3, 3 passes | always 9 dims, 1 pass |\n   340→| Standard | dims 1-5, 5 passes | dims 1-5, 5 passes | dims 1-5, 5 passes | dims 1-5, 5 passes | dims 1-5, 5 passes | always 9 dims, 1 pass |\n   341→| Deep | dims 1-5, 7 passes | dims 1-5, 7 passes | dims 1-5, 7 passes | dims 1-7, 7 passes | dims 1-5, 7 passes | always 9 dims, 1 pass |\n   342→\n   343→Pass counts are FIXED per depth. Quick = 3 passes, standard = 5 passes, deep = 7 passes. No extension. No early-exit. Final review is always a single scored pass across all 9 dimensions -- it is a gate, not a loop.\n   344→\n   345→---\n   346→\n   347→## Loop Cap Configuration\n   348→\n   349→The `workflow_policy` section of `run-config.yaml` (legacy: `phase_policy`) controls which workflows are enabled and sets an absolute safety ceiling per workflow. Only two fields exist: `enabled` and `loop_cap`. There is no `passes` field -- depth determines pass counts (3/5/7), not workflow policy.\n   350→\n   351→```yaml\n   352→workflow_policy:\n   353→  # Clarifier phase workflows\n   354→  discover:       { enabled: true, loop_cap: 10 }\n   355→  clarify:        { enabled: true, loop_cap: 10 }\n   356→  specify:        { enabled: true, loop_cap: 10 }\n   357→  spec-challenge: { enabled: true, loop_cap: 10 }\n   358→  author:         { enabled: false, loop_cap: 10 }\n   359→  design:         { enabled: true, loop_cap: 10 }\n   360→  design-review:  { enabled: true, loop_cap: 10 }\n   361→  plan:           { enabled: true, loop_cap: 10 }\n   362→  plan-review:    { enabled: true, loop_cap: 10 }\n   363→  # Executor phase workflows\n   364→  execute:        { enabled: true, loop_cap: 10 }\n   365→  verify:         { enabled: true, loop_cap: 5 }\n   366→  review:         { enabled: true, loop_cap: 10 }\n   367→  learn:          { enabled: true, loop_cap: 5 }\n   368→  prepare_next:   { enabled: true, loop_cap: 5 }\n   369→  run_audit:      { enabled: false, loop_cap: 10 }\n   370→```\n   371→\n   372→**`loop_cap`** is an absolute safety ceiling that prevents runaway loops regardless of depth. It is checked by `wazir capture loop-check` in pipeline mode. It is NOT the same as pass count (which is determined by depth: 3/5/7). Example: depth=deep gives 7 passes, but if `loop_cap: 5`, the cap guard fires at pass 5 and escalates. This is intentional -- the operator can constrain expensive phases.\n   373→\n   374→**Adaptive workflows** (`author`, `run_audit`) default to `enabled: false`. They are activated by explicit operator config or intent detection.\n   375→\n   376→**Post-run workflows** (`learn`, `prepare_next`) default to `enabled: true`. They run as part of the Final Review phase:\n   377→\n   378→- `learn` extracts durable learnings from review findings -- recurring findings become accepted learnings.\n   379→- `prepare_next` prepares context and handoff for the next run.\n   380→- `author` has a human approval gate, not an iterative review loop.\n   381→- `run_audit` is an on-demand standalone audit, not part of the main pipeline flow.\n   382→\n   383→---\n   384→\n   385→## Reviewer Mode Table\n   386→\n   387→The reviewer skill operates in different modes depending on the phase. **Mode is always explicit** -- the caller passes `--mode <mode>`. There is no auto-detection based on artifact availability.\n   388→\n   389→| Mode | Invoked during | Prerequisites | Dimensions | Output |\n   390→|------|---------------|---------------|------------|--------|\n   391→| `final` | After execution + verification | Completed task artifacts in `.wazir/runs/latest/artifacts/` | 9 final-review dims, scored 0-90 | Verdict: PASS/NEEDS FIXES/NEEDS REWORK/FAIL |\n   392→| `spec-challenge` | After specify | Draft spec artifact | 5 spec/clarification dims | Findings with severity, no score |\n   393→| `design-review` | After design approval | Design artifact, approved spec, accessibility guidelines | 5 design-review dims (canonical) | Findings with severity (blocking/advisory) |\n   394→| `plan-review` | After planning | Draft plan, approved spec, design artifact | 7 plan dims | Findings with severity, no score |\n   395→| `task-review` | During execution, per task | Uncommitted changes (or committed with known base SHA) | 5 task-execution dims | Pass/fail per task, no score |\n   396→| `research-review` | During discover | Research artifact | 5 research dims | Findings with severity, no score |\n   397→| `clarification-review` | During clarify | Clarification artifact | 5 spec/clarification dims | Findings with severity, no score |\n   398→\n   399→If `--mode` is not provided, the reviewer asks the user which review to run. This is the Netflix cross-incident pattern in embryonic form.\n\n---\n\n## What Wazir Already Has\n\nExamining the codebase, Wazir has significant infrastructure already in place:\n\n| Step | Status | Location |\n|------|--------|----------|\n| Review findings generated | Done | `docs/reference/review-loop-pattern.md` -- full review loop with dimension sets, scoring, Codex integration |\n| Findings logged per run | Done | `.wazir/runs/<id>/reviews/` -- per-pass findings files |\n| Cumulative findings appended | Specified | `workflows/learn.md` step 3 -- append to `memory/findings/cumulative-findings.md` |\n| Antipattern trigger check | Specified | `workflows/learn.md` step 4 -- every 5th run, prompt for review |\n| Antipattern catalog | Done | `expertise/antipatterns/` -- 22+ antipatterns across 7 categories |\n| Antipatterns loaded into reviewer | Done | `skills/reviewer/SKILL.md` -- expertise files are loaded into reviewer context |\n| Learning proposal template | Done | `templates/artifacts/learning-proposal.md` and `.template.json` |\n| Learner role | Done | `roles/learner.md` |\n\n**The gap:** Steps 3 and 4 in `workflows/learn.md` are specified but there is no structured mechanism to:\n- Classify findings by pattern (so recurrences can be counted)\n- Automatically detect recurrence (the 3+ threshold)\n- Generate an antipattern draft from clustered findings\n- Validate that the new antipattern actually reduces recurrence\n\n---\n\n## Minimum Viable Feedback-to-Learning Pipeline\n\n### Design Principles (from the research)\n\n1. **CodeGuru principle:** Make feedback zero-friction. Findings already exist -- don't require extra classification effort from the reviewer.\n2. **LangMem principle:** Let an LLM synthesize the pattern. Don't require humans to manually categorize findings into antipattern buckets.\n3. **Superpowers principle:** The antipattern must be grounded in observed failures, not hypothetical ones. The rationalization table is the proof it works.\n4. **Netflix principle:** Cross-run analysis is what surfaces systemic patterns. Individual run findings are necessary but not sufficient.\n\n### The Seven-Step Loop (Revised)\n\n```\n1. CAPTURE    Review finding logged with category tag\n2. ACCUMULATE Finding appended to cumulative-findings.md  \n3. CLUSTER    Findings grouped by semantic similarity (LLM-assisted)\n4. THRESHOLD  Cluster with 3+ findings across 2+ runs triggers promotion\n5. DRAFT      Antipattern drafted from cluster evidence\n6. INJECT     Antipattern loaded into reviewer expertise\n7. VALIDATE   Finding frequency in that cluster tracked -- should decline\n```\n\n### What to Ship: Steps 1-5 (No Full System Required)\n\n**Step 1: Structured Finding Capture** (enhancement to existing review loop)\n\nAdd a `category` field to each finding in the review pass log. This is the LangMem gradient principle: use an LLM to find the pattern across individual instances. Free-text categories clustered by LLM similarity are more robust. The taxonomy emerges from the data.\n\n**Why 3 findings / 2 runs as threshold?** Netflix and Google both find that cross-incident review is where systemic patterns emerge. A single run's findings might be contextual. Two runs with the same pattern is signal. Three findings is enough evidence to draft a concrete antipattern. The approval gate catches false positives.\n\n**Why project-scoped antipatterns (`memory/antipatterns/`) separate from global (`expertise/antipatterns/`)?** Project-specific patterns (e.g., \"this codebase's API client always forgets to refresh tokens\") should not pollute the global expertise library. Promotion from project to global is a separate, deliberate decision.\n\n**Why user approval at step 4, not full automation?** The Superpowers research is clear: you must observe the failure before codifying the fix. Automated promotion risks encoding noise as signal. The human gate at promotion is cheap insurance -- the automation handles everything up to that point.\n\n---\n\n## Summary\n\nThe minimum viable loop is **Steps 1-4**: structured capture, cumulative accumulation, LLM-assisted clustering, and threshold-triggered promotion with user approval. Steps 5-7 add value but are not required for the loop to start producing actionable learning.\n\nThe core insight across all four researched systems: **the hard part is not generating findings (Wazir already does that) or acting on patterns (the expertise system already does that).
